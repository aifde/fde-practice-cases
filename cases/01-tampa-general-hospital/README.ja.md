## 業務上の課題

Tampa Generalは大規模な大学病院システムを運営しています。患者配置、病床、人員、画像検査、退院、急性期医療の判断には、複数の臨床・運用システムに分散した情報が必要です。リスク予測だけでは業務は変わりません。結果を適切な時点で適切な担当者に届け、安全に判断できる文脈を添える必要があります。

TGHは2021年にPalantirとの協働を始めました。初期は運用データを接続して患者フローを支援し、その後AIを用いた診療連携基盤へ拡大しました。TGHによると、24時間稼働するCare Coordination Centerでは、Tilesと呼ばれる20件を超えるアプリケーションを利用しています。

## 導入の進め方

チームは汎用的な病院チャットボットから始めませんでした。PalantirのエンジニアとTGHの運用、分析、臨床の専門家が、患者をどこに配置するか、どの退院障壁を先に解消するか、勤務中の人員をどう調整するかといった具体的な判断を選びました。

基盤はリアルタイム運用データ、予測分析、Palantir AIP、大規模言語モデルを接続します。AIは優先順位、要約、判断材料を提供しますが、診療判断の責任はTGHの臨床・運用チームに残ります。

~~~mermaid
flowchart LR
  A[臨床・運用システム] --> B[患者と病床の共通モデル]
  B --> C[予測分析とAI業務]
  C --> D[配置、人員、退院、敗血症のキュー]
  D --> E[臨床・運用担当者の確認]
  E --> F[本番の診療業務で実行]
  F --> G[結果と業務フィードバック]
  G --> B
~~~

## システムと人による統制

| 層 | 公開情報で確認できる内容 |
| --- | --- |
| データ | 患者フロー、病床、人員、診療データを共通の運用画面へ集約する。 |
| 判断支援 | 患者配置、退院障壁、画像検査の滞留、人員需要、リスク信号を表示する。 |
| 人による統制 | 臨床・運用チームが情報を確認し、診療判断に責任を持つ。 |
| 展開 | 単一業務から10件を超える業務へ広げ、診療連携の運用基盤へ発展させた。 |
| フィードバック | 現場利用と運用結果を、次のアプリケーションと業務変更に反映する。 |

難所はモデルAPIではありません。データ接続、病院固有の判断規則、指令室への組み込み、高圧な現場での利用定着を同時に扱う必要がありました。

## 公開された成果

TGHは2024年6月に次の成果を公表しました。

- 患者配置に必要な時間を83%削減。
- 麻酔後ケアユニットでの滞留を28%削減。
- 敗血症患者の平均在院日数を30%削減。
- 1件の用途から病院全体の10件を超える業務へ拡大。

現在のCare Coordination Centerページでは、システム全体で4,000万ドルの費用削減、2万日の不要な入院日数の削減、救急外来の受入停止を25%削減、物理的な増床なしで30床分の実効容量を得たと説明しています。

数値はTGHとPalantirの公表値です。完全な実験設計や信頼区間は公開されておらず、全成果をAI基盤だけに帰属させることはできません。

## AI FDE事例として確信度が高い理由

1. エンジニアが高リスクの現場に入り、専門家と共同で作った。
2. 汎用AIではなく具体的な判断を対象にした。
3. 本番システムと既存業務へ接続した。
4. 完了条件をコードの出荷ではなく、現場利用と運用結果に置いた。
5. 最初の導入が、後続用途のデータと業務基盤になった。

## 未確認の点

- AI、業務再設計、人員変更、指令室の他技術の寄与は分離されていません。
- モデル評価、失敗率、アラート精度、患者属性別の性能は非公開です。
- 運用成果は参加組織の公表値であり、独立した臨床研究ではありません。
