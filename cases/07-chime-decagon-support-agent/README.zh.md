## 业务问题

Chime 需要在扩大会员客服规模的同时保持准确性、可靠性、安全和以会员为中心的体验。工作横跨聊天和电话，部分请求涉及敏感账户信息，以及补卡、查询入账状态等操作。

如果聊天和语音分别使用不同自动化系统，知识、政策和反馈会重复且不一致。Chime 希望用一个运营模式解决结构化请求，同时让人工和技术团队继续控制复杂案件和核心业务逻辑。

## 交付过程

Chime 使用回答准确性、自动化率、运营效率和客服净推荐值评估多个合作方，最终选择 Decagon 同时负责聊天和语音。

上线后的设计在不同渠道共享知识和行为层。聊天中的经验可以改进语音，语音反馈也能反向更新聊天。客户体验团队通过自然语言形式的 Agent Operating Procedures 调整代理行为，Chime 技术团队则保留核心逻辑和会员体验控制。

Decagon 表示，其团队直接与 Chime 产品和运营团队合作，持续修改行为和用例，并把会话洞察反馈给客服和产品团队。

~~~mermaid
flowchart LR
  A[聊天或语音会员请求] --> B[身份与账户上下文]
  B --> C[共享 Decagon 代理层]

  D --> E{解决还是升级}
  E -->|解决| F[获批回答或账户操作]
  E -->|升级| G[携带上下文转人工客服]
  F --> H[解决率与满意度信号]
  G --> H
  H --> I[业务行为更新与工程护栏]
  I --> C
~~~

## 系统与控制设计

| 问题 | 公开资料中的做法 |
| --- | --- |
| 渠道一致性 | 聊天和语音共享知识、会话上下文与改进信号。 |
| 业务规则 | 运营团队描述代理行为，技术团队控制核心逻辑。 |
| 人工接管 | 自动路径无法处理的案件携带上下文转交人工。 |
| 质量 | Chime 同时评估准确性、解决率、效率和会员满意度。 |
| 持续改进 | 产品和运营团队与 Decagon 共同调整用例与行为。 |

## 公开结果

Decagon 的 Chime 案例披露：

- 聊天解决率超过 70%。
- 语音解决率接近 70%。
- 首两周自动处理数十万条消息。
- 每月处理超过 100 万通电话，未报告可靠性问题。
- 相比其他候选供应商，会员满意度有所提升。

Decagon 的其他材料还披露客服成本下降 60%、会员满意度翻倍。这些数字没有公开测量周期与方法，因此只能作为供应商披露，不能视为独立验证。

## 为什么属于高置信度 AI FDE 案例

项目涉及受监管业务、多个渠道、账户上下文、业务操作、工程护栏和持续反馈。职责划分也很清楚：客户体验团队定义行为，工程师掌握硬性控制和集成，交付方帮助双方把政策转成生产系统。

## 尚未证明的部分

- Chime 没有发布独立技术报告。
- 解决率定义、评估样本和升级分布没有公开。
- “未出现可靠性问题”是客户案例表述，不是可用性报告。
- 成本和满意度数字缺少足够细节，无法与其他系统直接比较。
