## 業務上の課題

Oracle Fusion Applicationsは多数の業務機能を持ち、14,000社を超える顧客に利用されています。この環境へ生成AIを追加する作業は、単一機能ではなく製品群全体の課題です。各機能は正しい事業文脈へアクセスし、一貫したセキュリティとデータ境界を守り、用途に適したモデル挙動を持ち、企業顧客が採用できるリリース経路を必要とします。

互いに分断されたチャットボットを作ると、検索、ガバナンス、評価、統合の作業が重複します。OracleにはERP、HCM、供給網、営業、マーケティング、サービスにまたがる多数の小さな業務機能を支える共通アーキテクチャが必要でした。

## 導入の進め方

OracleとCohereはCommand R、Command R+、Embed、Rerankを組み合わせました。RAGによって企業情報を回答根拠に使い、裏付けのない出力を減らします。Embedが意味検索を支え、Rerankが候補の順序と関連性を改善します。

両社のチームは100件を超える生成AI用途で協働しました。例には財務説明文の生成、プロジェクト要約、サプライヤー推薦、交渉要約、候補者支援、サービスチャット要約、営業とマーケティングの文書作成があります。機能は既存のFusion画面へ組み込まれ、Oracleの通常の四半期更新で提供されました。

  E --> F[Fusionへ組み込まれた機能]
  F --> G[業務利用者の確認と操作]
  G --> H[四半期リリースと製品群のフィードバック]
  H --> B

## 共通の本番アーキテクチャ

| 要件 | 公開資料で説明された方法 |
| --- | --- |
| 根拠付け | RAGが企業アプリの文脈とCohereの生成を組み合わせる。 |
| 検索品質 | EmbedとRerankが意味検索と結果の順序を改善する。 |
| データ境界 | OracleアプリとOCIのセキュリティモデル内で動作し、顧客がデータガバナンスを管理する。 |
| 製品統合 | 財務、人事、供給網、営業、マーケティング、サービスの業務内にAIを表示する。 |
| 再利用 | 共通モデルと検索部品が、一つの専用アプリではなく広い製品群を支える。 |
| リリース | Fusion Applicationsの四半期更新で機能を改善する。 |

## 公開された成果

Cohereが2024年9月に公開した顧客事例は、Fusion Applications全体で100件を超える生成AI用途を支援したと報告しました。Oracleの後続資料は、数百のAI機能、100件を超える生成AI用途、50件を超えるエージェント型業務を説明しています。

公開資料には多くの機能例がありますが、統制された事業成果の測定値は限られています。最も強い根拠は単一の生産性向上率ではなく、公開済み統合の広さとOracleの製品資料に機能が記載されていることです。

## AI FDE事例として確信度が高い理由

現場エンジニアリングがプラットフォームエンジニアリングへ発展する姿を示します。多数の業務要件を、再利用可能なアーキテクチャ、ガバナンス、リリース機構へ変換しました。モデル選択、検索、セキュリティ、アプリ設計、顧客データ境界、継続的な製品提供を横断しています。成果は一度限りの概念実証ではなく、継続する企業向け製品の一部になりました。

## 未確認の点

- 用途の件数だけでは、各機能の利用と価値を示せません。
- 共通評価フレームワーク、機能別誤り率、顧客別利用状況は公開されていません。
- 一部は自律エージェントではなく生成支援機能です。
- 速度と正確性の改善は、一般に統制された数値なしで説明されています。
